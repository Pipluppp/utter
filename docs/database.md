This doc is a map. The SQL files are the real source of truth.

## Core Tables

| Table | Purpose |
| --- | --- |
| `profiles` | user profile, credits balance, remaining trials |
| `voices` | cloned and designed voice records plus qwen provider metadata |
| `generations` | generated audio records and provider output metadata |
| `tasks` | durable async job state for generate and design preview |
| `credit_ledger` | immutable credit events |
| `trial_consumption` | trial use and restore records |
| `billing_events` | Stripe webhook event log and processing state |

## Access Model

- `profiles`, `voices`, `generations`, and `tasks` use RLS.
- User-scoped reads rely on `auth.uid()`.
- Server-owned writes use the service role and RPCs.
- `credit_ledger`, `trial_consumption`, and `billing_events` are service-role managed. Authenticated users only get the access explicitly granted.

## Storage Model

- Buckets: `references`, `generations`
- Object keys are user-scoped
- DB rows store object keys, not permanent public URLs
- Signed URLs are generated by the API Worker

## Credit And Trial Model

Key SQL artifacts:

- `credit_apply_event(...)`
- `credit_usage_window_totals(...)`
- `trial_or_debit(...)`
- `trial_restore(...)`

Runtime meaning:

- new profiles default to `credits_remaining = 0`, `design_trials_remaining = 0`, `clone_trials_remaining = 0`
- generate charges credits through the ledger
- design preview uses a trial first only when the profile already has remaining design trials; otherwise it debits credits
- clone uses a trial first only when the profile already has remaining clone trials; otherwise it debits credits
- billing webhook grants prepaid credits into the ledger

## Tests

## Invariants

- Supabase stays the source of truth for task, voice, generation, credit, and billing state.
- User isolation depends on RLS and narrow grants, not frontend trust.
- Ledger writes must be idempotent.
- Trial restore paths must stay safe under retries and cancellations.
