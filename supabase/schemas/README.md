# Declarative Schemas

This directory contains declarative SQL schema files that describe the desired end-state of the database. They are the **authoring** source of truth for DDL going forward. The `supabase/migrations/` directory remains the **deployment** source of truth — migrations are what actually get applied to production.

## How It Works

This project uses a hybrid model:

- **Schema files** (`supabase/schemas/*.sql`) — You edit these to make DDL changes. They describe *what the database should look like*.
- **Migrations** (`supabase/migrations/*.sql`) — Generated by `supabase db diff`. They describe *how to get there*. Applied sequentially to production.

When you change a schema file and run `supabase db diff -f <name>`, the CLI spins up a shadow database, loads the schema files into it, compares against the cumulative migration state, and outputs an incremental migration capturing the delta.

You never write migrations by hand for DDL changes anymore. You edit schema files, diff, review, commit.

## File Layout

| File | Contents |
|------|----------|
| `00_extensions.sql` | `CREATE EXTENSION IF NOT EXISTS "pgcrypto"` — must come first, other objects depend on `gen_random_uuid()` |
| `01_tables.sql` | All 8 `CREATE TABLE` statements (profiles, voices, generations, tasks, credit_ledger, trial_consumption, billing_events, rate_limit_counters) with columns, constraints, defaults, foreign keys |
| `02_indexes.sql` | All `CREATE INDEX` statements (~18 indexes) |
| `03_functions.sql` | All 7 `CREATE OR REPLACE FUNCTION` statements (handle_updated_at, handle_new_user, credit_apply_event, credit_usage_window_totals, trial_or_debit, trial_restore, rate_limit_check_and_increment) |
| `04_triggers.sql` | `CREATE TRIGGER` statements (set_updated_at on profiles/tasks, on_auth_user_created on auth.users) |
| `05_rls_policies.sql` | `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` + all `CREATE POLICY` statements for public schema tables |
| `06_storage_policies.sql` | `CREATE POLICY` statements for `storage.objects` — legacy artifacts from before the Cloudflare R2 migration, not used by runtime |
| `07_grants.sql` | All `REVOKE` and `GRANT` statements for tables, sequences, and functions |

Files are ordered by dependency: extensions → tables → indexes → functions → triggers → policies → grants.

## Workflows

### Making Any Schema Change

1. Edit the appropriate schema file in `supabase/schemas/`.
2. Run `supabase db diff -f <descriptive_name>` to generate a migration.
3. Review the generated migration in `supabase/migrations/`. Pay attention to:
   - Grant duplication (known diff tool quirk — deduplicate if needed)
   - Policy drop+recreate behavior (expected, not a bug)
4. Run `supabase test db` to verify pgTAP tests still pass.
5. Commit both the schema file edit and the generated migration.

### Adding a New Table

1. Add the `CREATE TABLE` statement to `01_tables.sql`.
2. Add any indexes to `02_indexes.sql`.
3. Add `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` and `CREATE POLICY` statements to `05_rls_policies.sql`.
4. Add `REVOKE` and `GRANT` statements to `07_grants.sql`.
5. Run `supabase db diff -f add_<table_name>`.
6. Review the generated migration, then commit everything.

### Modifying a Function

1. Edit the function definition in `03_functions.sql`. Keep using `CREATE OR REPLACE FUNCTION`.
2. Run `supabase db diff -f update_<function_name>`.
3. Review and commit.

### Adding or Changing an RLS Policy

1. Edit `05_rls_policies.sql` — add a new `CREATE POLICY` or modify an existing one.
2. Run `supabase db diff -f <policy_change_name>`.
3. Review the migration. If you modified an existing policy, the diff will contain `DROP POLICY` + `CREATE POLICY` (not `ALTER POLICY`). This is expected — the diff tool does not support `ALTER POLICY`. There will be a brief window during migration where the policy does not exist (safe-by-default: access is denied when no policy exists).
4. Commit both files.

## What Must NOT Go in Schema Files

Schema files are DDL only. The following do not belong here:

| Prohibited | Why | Where It Goes Instead |
|------------|-----|----------------------|
| `INSERT` / `UPDATE` / `DELETE` statements | DML is not captured by `supabase db diff` — it will be silently ignored | Write a separate imperative migration: `supabase migration new <name>`, then add DML to the generated file |
| Storage bucket creation (`INSERT INTO storage.buckets`) | Bucket creation is DML | Imperative migration |
| Data backfills (e.g., populating a new column) | DML | Imperative migration |
| Seed data | DML | `supabase/seed.sql` or imperative migration |

If a future change requires both DDL and DML (e.g., adding a column then backfilling it), split it: edit the schema file and diff for the DDL part, then write a separate imperative migration for the DML part.

## Known Diff Tool Caveats

The diff tool (`migra` under the hood) has limitations. Be aware of these when reviewing generated migrations:

1. **RLS policies: drop+recreate only.** There is no `ALTER POLICY` support. Any policy modification produces `DROP POLICY` + `CREATE POLICY`. This means a brief window where the policy does not exist during migration. Since RLS defaults to deny-all when no policy exists, this is safe but worth knowing.

2. **Grant duplication from default privileges.** The diff tool may generate duplicate `GRANT` statements that are already covered by default privileges. Review and deduplicate grants in generated migrations before committing.

3. **DML not captured.** `INSERT`, `UPDATE`, `DELETE` statements in schema files are silently ignored by the diff. They will not appear in generated migrations. This is why DML is prohibited in schema files.

4. **View ownership edge cases.** View ownership changes may not be tracked correctly. If you work with views, manually verify ownership in the generated migration.

5. **Schema-level privileges not tracked.** `GRANT USAGE ON SCHEMA` and similar schema-level privileges are not captured by the diff tool.

6. **Comments, partitions, and domains not tracked.** `COMMENT ON`, partitioned table details, and `CREATE DOMAIN` are not reliably diffed.

7. **Column-level privileges.** Column-level `GRANT`/`REVOKE` may not be captured. Use table-level grants where possible.

8. **Materialized views.** Changes to materialized views may not diff correctly. Verify manually.

## schema_paths in config.toml

The `[db.migrations]` section in `supabase/config.toml` tells the CLI which schema files to use and in what order:

```toml
[db.migrations]
schema_paths = [
  "./schemas/00_extensions.sql",
  "./schemas/01_tables.sql",
  "./schemas/02_indexes.sql",
  "./schemas/03_functions.sql",
  "./schemas/04_triggers.sql",
  "./schemas/05_rls_policies.sql",
  "./schemas/06_storage_policies.sql",
  "./schemas/07_grants.sql"
]
```

Key points:

- Paths are relative to the `supabase/` directory.
- This is an **explicit list, not a glob**. If you add a new schema file, you must add it to this list.
- Order matters. Files are loaded in the order listed. Objects must be created before objects that depend on them (tables before indexes, functions before triggers, etc.). The numeric prefixes enforce this.
- The numeric prefix convention (`00_`, `01_`, ...) keeps lexicographic order aligned with dependency order.

## For AI Agents

If you are an AI agent working on this codebase:

- **To make a DDL change**: edit the appropriate schema file, then tell the user to run `supabase db diff -f <name>`. Do not write migration files by hand.
- **To find the current schema**: read the files in this directory. They represent the desired end-state of the database.
- **To understand what exists**: `01_tables.sql` has all tables, `03_functions.sql` has all functions, `05_rls_policies.sql` has all RLS policies, `07_grants.sql` has all security grants.
- **Do not add DML** (INSERT/UPDATE/DELETE) to any file in this directory. If you need DML, create a separate migration file in `supabase/migrations/`.
- **Do not use `IF NOT EXISTS`** on `CREATE POLICY` statements. The diff tool expects plain `CREATE POLICY`.
- **Always use `CREATE OR REPLACE FUNCTION`** for function definitions.
- **If you add a new schema file**, add it to `schema_paths` in `supabase/config.toml` in the correct dependency order.
- **Storage policies** in `06_storage_policies.sql` are legacy (runtime uses Cloudflare R2). Do not remove them unless explicitly migrating away from Supabase Storage.
